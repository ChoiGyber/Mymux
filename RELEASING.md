# Releasing Mymux

Mymux ships as a desktop app with **auto-update**: an installed app checks
GitHub Releases on startup and reveals an **⬆ 업데이트** button when a newer,
correctly-signed version exists.

- **Windows / Linux** installers are built automatically by GitHub Actions
  (`.github/workflows/release.yml`) when you push a version tag.
- **macOS** is built and uploaded manually from a Mac (see below).

## 0. Keep versions in sync

Every release **must** use the same version in all of these:

| File | Field |
| --- | --- |
| `crates/mycli-desktop/tauri.conf.json` | `"version"` |
| `crates/mycli-desktop/Cargo.toml` | `version` |
| git tag | `vX.Y.Z` |

The auto-updater compares the running app's version against `latest.json`, so a
mismatch breaks update detection.

## 1. One-time setup — signing secrets (required)

Auto-update artifacts are signed with a minisign key. The **public** key is
already committed in `tauri.conf.json` (`plugins.updater.pubkey`). The
**private** key must be stored as GitHub repository secrets so the workflow can
sign builds.

Set these two secrets (Repo → Settings → Secrets and variables → Actions), or
via the CLI after `gh auth login`:

```bash
gh secret set TAURI_SIGNING_PRIVATE_KEY < mymux_updater.key
gh secret set TAURI_SIGNING_PRIVATE_KEY_PASSWORD --body ""
```

- `TAURI_SIGNING_PRIVATE_KEY` — the full contents of the private key file.
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — the key password (empty for this key).

> ⚠️ Keep the private key safe and backed up. If it is lost, you can no longer
> sign updates and installed apps will stop accepting new versions.

## 2. Release Windows / Linux (automatic)

```bash
# After bumping the version in the two files above and committing:
git tag v0.2.0
git push origin v0.2.0
```

The workflow builds `.msi` / `.exe` (Windows) and `.deb` / `.AppImage` (Linux),
signs the updater artifacts, generates `latest.json`, and publishes a **draft**
GitHub Release. Review it, then click **Publish**.

## 3. Release macOS (manual, on a Mac)

The Mac build is produced on a Mac and added to the **same** GitHub Release that
the Windows/Linux workflow created. It must be signed with the **same** updater
key, or Mac users' auto-update will reject it.

### 3.1 Prerequisites (first time only)

```bash
# Xcode Command Line Tools
xcode-select --install

# Rust
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"

# Universal build targets (Apple Silicon + Intel)
rustup target add aarch64-apple-darwin x86_64-apple-darwin

# Tauri CLI v2
cargo install tauri-cli --version "^2"
```

### 3.2 Get the exact tagged source

Build the same commit as the release so the version matches:

```bash
git clone https://github.com/ChoiGyber/Mymux.git
cd Mymux
git checkout v0.2.0
```

### 3.3 Bring the signing key to the Mac

Copy `mymux_updater.key` from the machine where it was generated, or recreate it
from the `TAURI_SIGNING_PRIVATE_KEY` secret value into a file (e.g.
`~/mymux_updater.key`). This is the same key used by the CI workflow — do not
generate a new one, or installed apps will reject the update.

### 3.4 Build

```bash
export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/mymux_updater.key)"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD=""   # this key has no password

cd crates/mycli-desktop
cargo tauri build --target universal-apple-darwin
```

`createUpdaterArtifacts: true` makes the build emit the updater archive and its
signature automatically. Output lands in the **workspace-root** `target/`:

```
target/universal-apple-darwin/release/bundle/
├─ dmg/Mymux_0.2.0_universal.dmg   # the installer users download
└─ macos/
   ├─ Mymux.app
   ├─ Mymux.app.tar.gz             # updater archive
   └─ Mymux.app.tar.gz.sig         # its signature (goes into latest.json)
```

### 3.5 Upload to the v0.2.0 release

GitHub → Releases → `v0.2.0` → **Edit** → add these assets:

- `Mymux_0.2.0_universal.dmg`
- `Mymux.app.tar.gz`

### 3.6 Add macOS to `latest.json` (enables Mac auto-update)

The `latest.json` generated by CI only has Windows/Linux. Add the two darwin
blocks (the universal archive serves both, so `url` and `signature` are
identical):

1. Download `latest.json` from the release.
2. Copy the signature: `cat target/universal-apple-darwin/release/bundle/macos/Mymux.app.tar.gz.sig`
3. Add to `platforms` (keep the existing windows/linux entries):

   ```json
   "darwin-aarch64": {
     "signature": "<contents of Mymux.app.tar.gz.sig>",
     "url": "https://github.com/ChoiGyber/Mymux/releases/download/v0.2.0/Mymux.app.tar.gz"
   },
   "darwin-x86_64": {
     "signature": "<same .sig contents>",
     "url": "https://github.com/ChoiGyber/Mymux/releases/download/v0.2.0/Mymux.app.tar.gz"
   }
   ```

4. Delete the old `latest.json` asset on the release and re-upload the edited one
   (same filename).

### 3.7 Publish

Review the draft release, then click **Publish release**.

### Gatekeeper note (no Apple signing)

Without Apple code-signing/notarization, first launch shows an "unidentified
developer" warning. Users bypass it via Finder **right-click → Open**, or:

```bash
xattr -dr com.apple.quarantine /Applications/Mymux.app
```

The updater's minisign verification is independent of Apple signing, so
auto-update still works. For a smoother UX, add Apple `codesign` + notarization
later with an Apple Developer account.

> Tip: if you'd rather have CI build macOS too (no manual signing/`latest.json`
> editing), add a `- platform: macos-latest` entry back to the matrix in
> `release.yml` with `args: "--target universal-apple-darwin"`.
